#Глава 1. Обзор архитектуры
## 1.1 Назначение проекта
PizzaSaga — это распределённая микросервисная система, демонстрирующая современные подходы к построению высоконагруженных backend-приложений на платформе .NET.

Основная цель проекта — показать практическое применение архитектурных паттернов, используемых в production-системах:
Clean Architecture (упрощённая);
Tactical DDD;
CQRS;
Saga Orchestration;
Transactional Outbox;
Event-Driven Architecture;
OpenTelemetry;
Distributed Tracing;
Resilience;
Health Checks;
централизованная наблюдаемость (Observability).

Проект демонстрирует решение типичных проблем распределённых систем:
согласованность данных;
гарантированную публикацию сообщений;
идемпотентную обработку повторно доставленных сообщений (паттерн Idempotent Consumer);
компенсацию ошибок;
трассировку запросов;
устойчивость к частичным отказам.

## 1.2 Архитектурная цель
В распределённой системе невозможно использовать одну общую транзакцию между несколькими сервисами и различными хранилищами данных.

Вместо этого проект строится вокруг следующих принципов:
каждый сервис полностью владеет своими данными;
взаимодействие между сервисами происходит посредством событий;
бизнес-процесс координируется Saga Orchestrator;
доставка сообщений гарантируется Transactional Outbox;
все операции являются наблюдаемыми благодаря OpenTelemetry.

Таким образом достигается eventual consistency без использования распределённых транзакций.

## 1.3 Основные архитектурные принципы
При разработке проекта используются следующие принципы.

Single Responsibility
Каждый сервис отвечает только за одну бизнес-область.

Например:
Order Service — жизненный цикл заказа;
Stock Service — управление остатками товаров;
Payment Service — резервирование и подтверждение оплаты;
Auth Service — аутентификация и выпуск JWT;
API Gateway — единая точка входа в систему.

Database per Service
Каждый микросервис использует собственное хранилище данных.
Сервисы никогда не обращаются напрямую к базам данных друг друга.

Optimistic Concurrency
Для предотвращения конфликтов при одновременном изменении данных используется оптимистичный контроль конкурентного доступа (Optimistic Concurrency).
При изменении агрегата Order проверяется версия записи. Если с момента её чтения данные были изменены другим процессом, операция завершается ошибкой конкурентного доступа, после чего может быть повторно выполнена с использованием актуального состояния сущности.
Например, во время выполнения Order Saga несколько сообщений могут почти одновременно попытаться изменить состояние одного и того же заказа. Проверка версии записи предотвращает потерю изменений и не допускает выполнения некорректных переходов между состояниями.
Использование Optimistic Concurrency позволяет избежать длительных блокировок базы данных и обеспечивает корректную обработку параллельных изменений в распределённой системе.

Обмен информацией осуществляется исключительно через:
HTTP API (при необходимости);
RabbitMQ;
события интеграции.

Это уменьшает связанность сервисов и позволяет независимо изменять внутреннюю модель данных.

Event-Driven Architecture
Основным способом взаимодействия между сервисами являются события.
После завершения бизнес-операции сервис публикует событие, которое может быть обработано другими сервисами.
Ни один сервис не управляет внутренним состоянием другого сервиса.

Eventual Consistency (согласованность в конечном счете)
Это модель согласованности данных в распределенных системах, при которой изменения не отображаются на всех узлах моментально, но гарантируется, что если новые обновления прекратятся, то спустя некоторое время все копии данных станут одинаковыми.
Система не стремится обеспечить мгновенную согласованность между всеми сервисами.
После выполнения локальной транзакции изменения постепенно распространяются через события.

Это позволяет:
избежать распределённых транзакций;
повысить отказоустойчивость;
обеспечить горизонтальное масштабирование.

Loose Coupling (слабая связанность)
Это архитектурный принцип в программировании, при котором компоненты системы слабо зависят друг от друга. Изменения в одном модуле не требуют доработки остальных, так как они взаимодействуют только через заранее определенные интерфейсы или контракты. Это повышает гибкость и упрощает поддержку кода.
Сервисы знают только о контрактах обмена сообщениями.

Они не зависят:
от внутренних моделей друг друга;
от структуры БД других сервисов;
от конкретной реализации бизнес-логики.

## 1.7 Основной бизнес-процесс
Основным сценарием системы является оформление заказа. Бизнес-процесс представляет собой последовательность локальных транзакций, выполняемых различными микросервисами и координируемых Order Saga.

Типичный сценарий успешного выполнения выглядит следующим образом:

HTTP POST /orders
        │
        ▼
OrderCreated (Domain Event)
        │
        ▼
		...
        │
        ▼
OrderCompletedEvent

Если один из этапов завершается неуспешно, Order Saga инициирует компенсирующие действия. Например, при ошибке резервирования оплаты ранее зарезервированный товар освобождается, после чего заказ переводится в состояние Cancelled.

Подробное описание жизненного цикла заказа, состояний Saga, используемых команд, интеграционных событий и компенсационных сценариев приведено в главе 7 «Saga и обмен сообщениями».

## 1.8 Наблюдаемость системы
Каждый запрос получает уникальный TraceId и CorrelationId.

Эти идентификаторы автоматически передаются:
между HTTP-запросами;
через RabbitMQ;
через MassTransit;
в журналы;
в трассировки OpenTelemetry;
в метрики.

Это позволяет полностью восстановить цепочку выполнения любого бизнес-процесса независимо от количества задействованных сервисов.

## 1.9 Почему именно такая архитектура
Цель проекта — не продемонстрировать максимальное количество технологий, а показать применение архитектурных решений, характерных для современных распределённых систем.

Выбранный подход обеспечивает:
независимость сервисов;
возможность горизонтального масштабирования;
высокую отказоустойчивость;
наблюдаемость всех компонентов системы;
гарантированную доставку сообщений;
поддержку длительных бизнес-процессов через Saga;
минимальную связанность между сервисами;
удобство дальнейшего развития проекта.

Каждый архитектурный паттерн используется только там, где он решает конкретную задачу. Это позволяет избежать избыточной сложности и сохранить архитектуру эволюционной, когда новые механизмы внедряются только при появлении соответствующих требований.

#2. Архитектурные принципы
Данная глава описывает архитектурные подходы, лежащие в основе проекта PizzaSaga.
Архитектура проекта не следует какой-либо одной методологии. Вместо этого используется комбинация нескольких взаимодополняющих подходов, каждый из которых решает определённую инженерную задачу.

Основными архитектурными принципами являются:
Clean Architecture;
Tactical Domain-Driven Design (DDD);
CQRS;
Vertical Slice Architecture;
Event-Driven Architecture.

Помимо перечисленных архитектурных подходов, в проекте применяются инженерные практики, обеспечивающие надёжность распределённой системы: Saga Orchestration, Transactional Outbox, Idempotent Consumers и Optimistic Concurrency. Они рассматриваются в отдельных главах документации, поскольку относятся не к организации кода, а к обеспечению согласованности данных и отказоустойчивости.

Использование нескольких подходов позволяет получить систему, которая остаётся простой для сопровождения, хорошо масштабируется и соответствует современным практикам разработки распределённых приложений на платформе .NET.

## 2.1 Clean Architecture
Все микросервисы проекта построены в соответствии с принципами Clean Architecture.
Основная цель — отделить бизнес-логику от инфраструктуры и внешних технологий.

В отличие от упрощённой реализации с разделением по папкам, в данном проекте используется физическое разделение слоёв на отдельные проекты.
Каждый слой имеет собственную область ответственности и компилируется в отдельную сборку.
Это позволяет контролировать зависимости между слоями не только архитектурными соглашениями, но и средствами компилятора.

Основной принцип
Все зависимости направлены внутрь приложения.

	Api
      │
      ▼
Application
      │
      ▼
	Domain
      ▲
      │
Infrastructure

Внутренние слои ничего не знают о внешних.

Domain не зависит от Application слое.
Application также не зависит от инфраструктурных технологий.
Благодаря этому бизнес-логика остаётся независимой от конкретных технологий хранения данных, обмена сообщениями и публикации API.

В PizzaSaga каждый слой представлен отдельным проектом.
Это позволяет:
контролировать зависимости на уровне компиляции;
уменьшить связанность компонентов;
упростить сопровождение;
облегчить модульное тестирование;
сделать архитектуру более явной.

Такой подход ближе к архитектуре, применяемой в крупных корпоративных проектах.

## 2.2 Tactical Domain-Driven Design
Проект использует тактические паттерны Domain-Driven Design.
Полная реализация стратегического DDD не требуется, поскольку предметная область проекта относительно компактна.
Используются только те элементы DDD, которые действительно повышают выразительность бизнес-модели.

Aggregate
Каждая бизнес-сущность представлена отдельным Aggregate Root.

Например:
Order
Inventory
PaymentReservation

Именно Aggregate управляет собственным состоянием.
Изменение состояния выполняется исключительно через его методы.
Aggregate также отвечает за соблюдение бизнес-инвариантов и обеспечивает консистентность данных внутри своих границ.

Например:
order.Cancel();
order.MarkInventoryReserved();
order.MarkPaymentReserved();

Внешний код не должен изменять состояние агрегата напрямую.
Если бизнес-логика не принадлежит конкретному Aggregate, допускается использование Domain Service.

Entity
Entity обладают собственной идентичностью и изменяемым состоянием. В большинстве случаев они входят в состав Aggregate.

Например:
Order
OrderItem
InventoryItem
PaymentReservation

Value Object
Value Object описывают значения, не имеющие собственной идентичности.

Например:
Money
Address
PhoneNumber
Email

Value Object являются неизменяемыми.
Равенство Value Object определяется их значением, а не ссылкой на объект.
Это уменьшает вероятность ошибок и делает бизнес-модель более предсказуемой.

Domain Events используются только внутри сервиса, подробнее см. главу 7.
Для взаимодействия между микросервисами используются интеграционные команды и интеграционные события, определённые в общем контракте сообщений.
Это позволяет разделить внутреннюю модель предметной области и внешний контракт взаимодействия.

Что сознательно не используется
В проекте намеренно отсутствуют следующие паттерны:
Generic Repository поверх Entity Framework Core;
Unit of Work поверх DbContext;
Service Locator;
Anemic Domain Model (анемичная модель предметной области) классы-сущности содержат только данные;
Factory используются только тогда, когда создание агрегата невозможно выразить обычным конструктором.

Отказ от этих паттернов обусловлен стремлением сохранить архитектуру максимально простой и выразительной.

## 2.3 CQRS
Command Query Responsibility Segregation (CQRS) — это архитектурный паттерн, который разделяет операции чтения и записи данных в приложении на две независимые модели.
В проекте используется упрощённая реализация Command Query Responsibility Segregation.

CQRS разделяет операции изменения данных и операции чтения.

Commands
↓
Business Logic
↑
Queries

Такое разделение делает бизнес-логику более понятной и облегчает дальнейшее развитие проекта.

Commands
Команда представляет собой запрос на изменение состояния системы.

Например:
CreateOrder
CancelOrder
ReserveInventory
ReservePayment

Каждая команда имеет единственный обработчик.

Обработчик выполняет:
валидацию;
выполнение бизнес-операции;
сохранение изменений.
После сохранения инфраструктурный слой публикует накопленные Domain Events.

Queries
Запрос не изменяет состояние системы.

Например:
GetOrder
GetOrderStatus
GetInventory

Запросы читают данные напрямую из базы данных.
При необходимости в будущем может быть выделена отдельная модель чтения.

В качестве реализации CQRS используется библиотека Mediator. Она обеспечивает статическую генерацию обработчиков через Source Generators, поддерживает Pipeline Behaviors и не требует регистрации обработчиков через Reflection, что упрощает навигацию по коду и делает решение совместимым с Native AOT.

Pipeline Behaviors
Повторяющаяся функциональность выносится в Pipeline Behaviors.

Например:
ValidationBehavior
LoggingBehavior
TransactionBehavior
PerformanceBehavior

Это позволяет оставить обработчики максимально компактными и сосредоточенными исключительно на бизнес-логике.

## 2.4 Vertical Slice Architecture
Внутри слоя Application используется подход Vertical Slice Architecture.
Архитектура вертикального среза Vertical Slice Architecture (VSA) — это подход к проектированию программных систем, при котором приложение делится на независимые функции (фичи), а не на технические уровни (слои).

В отличие от классического разделения по техническим слоям, код организуется вокруг бизнес-функций.

Например:

Application
└── Features
    └── Orders
        ├── Create
        ├── Cancel
        └── Get

Каждая бизнес-функция содержит все необходимые компоненты.

Например:
Create
CreateOrderCommand
CreateOrderHandler
CreateOrderValidator
CreateOrderRequest
CreateOrderResponse

Благодаря этому изменения оказываются локализованы внутри одной функциональной области и упрощает дальнейшее развитие системы.

Почему Vertical Slice используется только в Application?
Vertical Slice описывает прикладные сценарии использования системы.

Domain моделирует предметную область и не должен зависеть от конкретных пользовательских сценариев.

Поэтому Domain организуется по бизнес-сущностям:
Aggregates
Entities
ValueObjects
Domain Events

а Application — по сценариям использования.

Такое разделение позволяет совместить выразительную доменную модель с удобной организацией прикладного кода.

## 2.5 Event-Driven Architecture
Event-Driven Architecture (EDA или событийно-ориентированная архитектура) — это подход к проектированию ПО, при котором компоненты системы общаются не через прямые синхронные запросы (например, REST API), а через публикацию и обработку событий — сигналов об изменении состояния.
Межсервисное взаимодействие строится на асинхронного обмене событиями.
Каждый сервис публикует события о произошедших изменениях и реагирует на события других сервисов.

Такой подход обеспечивает:
слабую связанность сервисов;
независимое масштабирование;
отказоустойчивость;
поддержку длительных бизнес-процессов.

Для доставки сообщений используется RabbitMQ, а для управления бизнес-процессом — Saga Orchestrator на базе MassTransit.

## 2.7 Основные архитектурные принципы

При разработке всех сервисов проекта соблюдаются следующие правила:
бизнес-логика не зависит от инфраструктуры;
каждый Aggregate самостоятельно управляет своим состоянием;
команды и запросы имеют независимые обработчики;
каждая бизнес-операция реализуется отдельным обработчиком;
повторяющаяся функциональность выносится в Pipeline Behaviors;
взаимодействие между сервисами осуществляется только через публичные API или обмен сообщениями;
внутренние модели сервиса никогда не используются другими сервисами;
предпочтение отдаётся простым решениям, если они удовлетворяют требованиям системы;
интеграционные события публикуются только после успешного сохранения данных с использованием Transactional Outbox;
обработчики сообщений проектируются идемпотентными, чтобы повторная доставка сообщений не приводила к повторному выполнению бизнес-операций.

#3. Внешние интерфейсы системы
## 3.1. Назначение
Внешние интерфейсы определяют способы взаимодействия клиентов с системой независимо от внутренней реализации микросервисов.

Основным способом взаимодействия является HTTP REST API через API Gateway. В дальнейшем архитектура допускает использование дополнительных механизмов интеграции (WebSocket, Webhook, gRPC и др.) без изменения внутренних бизнес-процессов.
Для внешнего клиента система представляет собой единый API независимо от количества микросервисов, участвующих в обработке запроса.

## 3.2. Общая модель взаимодействия
Все клиентские запросы поступают через API Gateway.
Публичный API поддерживает версионирование. Все клиентские запросы выполняются через соответствующую версию API (например, /api/v1/orders), что позволяет развивать контракт без нарушения обратной совместимости.

Client
    │
    ▼
API Gateway
    │
    ▼
Микросервис
    │
    ▼
Business Logic

API Gateway отвечает за:
маршрутизацию запросов;
проверку JWT-токенов и передачу информации о пользователе во внутренние сервисы;
централизованную обработку версий API;
обеспечение обратной совместимости при развитии публичного API;
унифицированное представление ошибок (ProblemDetails);
распространение CorrelationId между сервисами;
сокрытие внутренней структуры микросервисов;
единый вход в систему.

Каждый микросервис предоставляет собственную OpenAPI-спецификацию (Swagger) для документирования своих REST API. API Gateway не агрегирует документацию сервисов, поскольку OpenAPI предназначена главным образом для разработки, тестирования и сопровождения отдельных сервисов.

Клиент не взаимодействует с микросервисами напрямую.

## 3.2.1 Контракты API
Все публичные REST API используют единые соглашения:
JSON в качестве основного формата обмена;
HTTP-коды состояния в соответствии со спецификацией REST;
ProblemDetails для представления ошибок;
единые правила именования ресурсов;
версионирование API.

Это обеспечивает единообразие взаимодействия клиентов со всеми сервисами системы.

## 3.3 Аутентификация
Перед обращением к защищённым методам клиент должен пройти аутентификацию и получить JWT-токен.
Для этого используется Auth Service.

Общий сценарий взаимодействия выглядит следующим образом.

POST /auth/login
        │
        ▼
Проверка пользователя
        │
        ▼
JWT Access Token
        │
        ▼
Client
        │
        ▼
Authorization: Bearer <token>
        │
        ▼
API Gateway

После успешной аутентификации клиент использует полученный JWT-токен при обращении ко всем защищённым REST API.
JWT-токен передаётся в заголовке Authorization по схеме Bearer. 
Проверка JWT выполняется централизованно в API Gateway. Внутренние сервисы не взаимодействуют с Auth Service напрямую при обработке каждого запроса.

## 3.4. Синхронные и асинхронные операции
В системе используются два типа операций.

Синхронные операции
Запрос завершается в рамках одного HTTP-вызова.

Примеры:
получение списка заказов;
получение информации о товаре;
изменение профиля пользователя;
получение статуса заказа.

Такие операции обычно возвращают код ответа:
200 OK;
201 Created;
204 No Content.

Асинхронные операции
Некоторые бизнес-процессы требуют взаимодействия нескольких микросервисов и не могут быть завершены за время одного HTTP-запроса.

Например:
оформление заказа;
возврат денежных средств;
создание доставки;
выполнение сложных бизнес-процессов.

В подобных случаях сервис:
принимает запрос;
сохраняет первоначальное состояние;
инициирует выполнение бизнес-процесса;
сразу возвращает ответ клиенту.

Клиент получает подтверждение о принятии запроса, а не о завершении бизнес-операции.
Сам процесс продолжается асинхронно.

## 3.5. Асинхронная модель обработки
Для длительных бизнес-процессов HTTP-запрос не ожидает завершения всех внутренних операций.
После получения запроса сервис создаёт новый ресурс и возвращает клиенту ответ 201 Created с его идентификатором и начальным состоянием Pending.
Общая схема обработки выглядит следующим образом.

POST /orders
        │
        ▼
201 Created
Status = Pending
        │
        ▼
GET /orders/{id}
        │
        ▼
Completed
POST /orders

Ответ:
201 Created
{
    "orderId": "8e24b5d2",
    "status": "Pending"
}

Возвращение ответа 201 Created означает успешное создание ресурса, но не завершение полного бизнес-процесса.
После возврата ответа дальнейшая обработка продолжается асинхронно до достижения конечного состояния заказа.

Например:
резервирование товара;
проведение оплаты;
создание доставки;
отправка уведомлений.

Итоговый результат будет сформирован позже после завершения всех этапов обработки.

## 3.6. Получение состояния бизнес-процесса
После создания ресурса клиент может получить его текущее состояние.

Например:
GET /orders/{id}

Ответ во время выполнения процесса:
{
    "orderId": "8e24b5d2",
    "status": "Pending"
}

После успешного завершения:
{
    "orderId": "8e24b5d2",
    "status": "Completed"
}

При невозможности завершить процесс:
{
    "orderId": "8e24b5d2",
    "status": "Cancelled"
}

Конкретный набор состояний определяется бизнес-логикой соответствующего сервиса. При этом клиент взаимодействует только с текущим состоянием ресурса, не завися от внутреннего механизма выполнения бизнес-процесса.

## 3.7. Способы уведомления клиента
Архитектура поддерживает несколько моделей получения результата выполнения асинхронных операций.

Polling (опрос состояния ресурса)
Наиболее распространённый вариант.
Именно этот способ используется в рамках данного проекта.
После получения идентификатора ресурса клиент периодически выполняет запрос:
GET /orders/{id}
до тех пор, пока процесс не перейдёт в конечное состояние.

Преимущества:
простая реализация;
отсутствие постоянного соединения;
подходит большинству клиентов.

Архитектура также допускает использование push-механизмов (например, WebSocket, SignalR или Server-Sent Events), а также Webhook при интеграции с внешними системами. В рамках текущего проекта данные способы уведомления не реализуются, однако публичный API не ограничивает их последующее добавление.

## 3.8. Независимость внешнего API от внутренней реализации
Контракт внешнего API не зависит от механизма реализации бизнес-процесса.

Внутри системы могут использоваться различные механизмы координации и обмена сообщениями, такие как Saga, Transactional Outbox, брокеры сообщений и компенсационные операции.

Для клиента все эти детали остаются скрытыми. Он взаимодействует только с публичным API и состоянием ресурсов.
Внутренние механизмы обмена сообщениями и координации между микросервисами рассматриваются в главе 7. Saga и обмен сообщениями.

#5. Структура одного сервиса
## 5.1 Общая организация
Каждый бизнес-сервис в системе имеет одинаковую внутреннюю организацию независимо от предметной области.

В качестве примера далее рассматривается Order Service, однако аналогичная структура используется и в остальных сервисах.

Каждый сервис построен в соответствии с принципами Clean Architecture и состоит из нескольких проектов, каждый из которых отвечает за собственную область ответственности.

Единая структура сервисов обеспечивает:
предсказуемую организацию кода;
единообразие разработки;
удобство сопровождения;
упрощение навигации по проекту.

#6. Зависимости между проектами
## 6.1 Назначение
Одним из ключевых принципов архитектуры PizzaSaga является строгое управление зависимостями между проектами.

Архитектурные ограничения должны обеспечиваться не только соглашениями команды, но и физической структурой Solution. Для этого каждый слой сервиса выделен в отдельный проект, а допустимые зависимости между ними заранее определены.

Такой подход позволяет сохранить независимость бизнес-логики от инфраструктуры, уменьшить связанность компонентов и упростить сопровождение системы.

## 6.2 Общая схема зависимостей
Ниже показаны допустимые зависимости между проектами одного сервиса.

                     +-------------------+
                     |     Order.Api     |
                     +---------+---------+
                               │
              ┌────────────────┴────────────────┐
              ▼                                 ▼
+---------------------------+     +---------------------------+
|   Order.Application       |<────|   Order.Infrastructure    |
+-------------+-------------+     +-------------+-------------+
              │                                 │
              └─────────────────┬───────────────┘
                                ▼
                     +----------------------+
                     |    Order.Domain      |
                     +----------------------+

Физические зависимости между проектами соответствуют принципам Clean Architecture.

Проект			Зависит от
Api				Application, Infrastructure
Application		Domain
Infrastructure	Application, Domain
Domain			—

## 6.3 Правило зависимостей
Главное правило архитектуры формулируется следующим образом:
Зависимости всегда направлены от внешних слоёв к внутренним.

Это означает:
внешние слои могут зависеть от внутренних;
внутренние слои никогда не зависят от внешних;
бизнес-логика остаётся независимой от инфраструктурных технологий.

Такое направление зависимостей соответствует принципу Dependency Inversion Principle (DIP) и позволяет свободно изменять инфраструктурные компоненты без влияния на предметную область.


## 6.4 Общие библиотеки
Помимо проектов конкретного сервиса, решение содержит две общие библиотеки:

PizzaSaga.Contracts — публичные контракты взаимодействия между микросервисами (команды, интеграционные события, общие DTO);
PizzaSaga.BuildingBlocks — переиспользуемые инфраструктурные компоненты и вспомогательные классы.

Микросервисы могут зависеть от этих библиотек, однако не имеют прямых ссылок друг на друга.

Infrastructure	✅		✅			—				❌	✅			✅

## 6.7 Архитектурные ограничения
При разработке сервисов соблюдаются следующие правила:

Domain не зависит от других проектов решения;
Application не использует инфраструктурные технологии напрямую;
Infrastructure реализует интерфейсы, объявленные в Application;

между сервисами отсутствуют прямые ссылки.

Автоматическая проверка предотвращает нарушение архитектурных правил по мере развития проекта.

#7. Saga и обмен сообщениями
## 7.1 Назначение
В системе PizzaSaga отсутствуют распределённые транзакции между сервисами.
Каждый микросервис владеет собственной базой данных и самостоятельно отвечает за согласованность своих данных.
Для координации длительных бизнес-процессов используется паттерн Saga, а обмен данными между сервисами осуществляется посредством асинхронных сообщений через RabbitMQ.

Такой подход позволяет:
исключить распределённые транзакции;
обеспечить слабую связанность сервисов;
повысить отказоустойчивость системы;
независимо масштабировать сервисы;
поддерживать длительные бизнес-процессы.

## 7.2 Общая схема взаимодействия
В качестве брокера сообщений используется RabbitMQ.
Для построения обмена сообщениями применяется библиотека MassTransit.

                HTTP
                  │
                  ▼
           PizzaSaga.ApiGateway
                  │
                  ▼
             Order.Api
                  │
                  ▼
        Order.Application
                  │
                  ▼
          Order State Machine
                  │
        ┌─────────┴─────────┐
        ▼                   ▼
Reserve Inventory     Reserve Payment
        │                   │
        ▼                   ▼
Stock Service        Payment Service
        │                   │
        └─────────┬─────────┘
                  ▼
          Order State Machine
                  │
                  ▼
            Order Completed

Что такое State Machine
State Machine (машина состояний) — это обычный класс, который хранит описание жизненного цикла Saga.
Выглядит примерно так:

Order.Application
│
├── Commands
├── Queries
├── Consumers
├── StateMachines
│      ├── OrderStateMachine.cs
│      └── OrderState.cs
└── ...

Где она находится
Она находится внутри Order Service:

               Order Service
+------------------------------------------+
|                API                       |
|                    │                     |
|                    ▼                     |
|             Application                  |
|                    │                     |
|                    ▼                     |
|         Order State Machine              |
|                    │                     |
|		Saga (MassTransit State Machine)   |
|      ┌─────────────┴─────────────┐       |
|      ▼                           ▼       |
| Publish ReserveInventoryCommand   Publish ReservePaymentCommand
+------------------------------------------+
               │                     │
               ▼                     ▼
        Stock Service        Payment Service

Order Saga является оркестратором бизнес-процесса, а Order Service является хостом этой Saga.
Остальные сервисы выполняют только собственную бизнес-операцию и не знают о существовании друг друга.

## 7.3 Почему используется Saga
Создание заказа включает несколько независимых операций:
создание заказа;
резервирование товаров;
резервирование оплаты;
подтверждение заказа.

Каждая операция выполняется как локальная транзакция соответствующего сервиса.

Использование распределённой транзакции Two-Phase Commit (2PC, двухфазный коммит) в подобной архитектуре приводит к высокой связанности сервисов, блокировке ресурсов до завершения операции, снижению производительности и усложнению масштабирования.
Поэтому вместо распределённой транзакции используется Saga, которая оперирует локальными транзакциями, обеспечивает согласованность в конечном итоге (eventual consistency) и не блокирует ресурсы глобально.
Saga представляет собой последовательность локальных транзакций, объединённых общим бизнес-процессом.
Каждая локальная транзакция либо подтверждает выполнение операции, либо инициирует компенсирующее действие.

## 7.4 Оркестратор Saga
В проекте применяется оркестрируемая Saga (Orchestrated Saga).
Оркестрируемая Saga (Orchestrated Saga) — это паттерн управления распределенными транзакциями в микросервисной архитектуре, где специальный центральный сервис (оркестратор) берет на себя координацию всех шагов бизнес-процесса, отправляя команды остальным сервисам и запуская компенсационные действия при сбоях.

Оркестратором является Order Service.
Он управляет последовательностью выполнения операций и принимает решения о переходе между состояниями.
Для реализации используется MassTransit State Machine.

Основные обязанности оркестратора:
создание экземпляра Saga;
управление состояниями заказа;
отправка команд другим сервисам;
обработка интеграционных событий;
запуск компенсирующих операций;
завершение бизнес-процесса.

Остальные сервисы не знают о существовании Saga и не взаимодействуют друг с другом напрямую.

Состояния экземпляра Saga (State Machine).
В процессе выполнения экземпляр Saga последовательно переходит между следующими состояниями:
                Initial
                    │
                    ▼
			AwaitingInventory
          │                 │
          │                 ▼
          │            Cancelled
          ▼
			AwaitingPayment
          │                 │
          │                 ▼
          │            Cancelled
          ▼
      Completed

В виде таблицы:
Состояние						Что означает
Initial							Начальное состояние до создания экземпляра Saga. При получении первого события создаётся новый экземпляр и выполняется переход в следующее состояние.
AwaitingInventoryReservation	Отправлена команда ReserveInventoryCommand, ожидается событие InventoryReservedEvent или InventoryReservationFailedEvent.
AwaitingPaymentReservation		Товар успешно зарезервирован, отправлена команда ReservePaymentCommand, ожидается PaymentReservedEvent или PaymentReservationFailedEvent.
Completed						Все этапы успешно завершены, заказ подтверждён.
Cancelled						Бизнес-процесс завершён с отменой заказа (из-за ошибки или после выполнения компенсации).

Экземпляр Saga сохраняется в отдельной записи базы данных. Помимо текущего состояния, он содержит данные, необходимые для продолжения выполнения бизнес-процесса после получения очередного сообщения.

## 7.5 Жизненный цикл заказа
Ниже приведён типичный сценарий успешного выполнения заказа.

Create Order (HTTP POST /orders)
        │
        ▼
OrderCreated (Domain Event)
        │
        ▼
ReserveInventoryCommand
        │
        ▼
InventoryReservedEvent
        │
        ▼
ReservePaymentCommand
        │
        ▼
PaymentReservedEvent
        │
        ▼
OrderCompleted (Domain Event)
        │
        ▼
OrderCompletedEvent (Integration Event)

После получения ожидаемых интеграционных событий Saga выполняет переход между состояниями. 
Доменные события используются внутри Order Service для изменения состояния заказа и публикации соответствующих интеграционных событий.
После успешного резервирования товара и оплаты Order Service переводит заказ в состояние Completed, генерирует доменное событие OrderCompleted, которое затем преобразуется в интеграционное событие OrderCompletedEvent.

## 7.6 Компенсирующие операции
Если одна из бизнес-операций завершается ошибкой, Saga запускает компенсирующие операции, позволяющие вернуть систему в согласованное состояние.
Компенсация представляет собой отдельную бизнес-операцию. Она не является откатом транзакции базы данных. Каждый микросервис самостоятельно отменяет изменения, которые ранее успешно выполнил.
Ниже приведён пример компенсации при ошибке резервирования оплаты.

Create Order
      │
      ▼
OrderCreated (Domain Event)
      │
      ▼
ReserveInventoryCommand
      │
      ▼
InventoryReservedEvent
      │
      ▼
ReservePaymentCommand
      │
      ▼
PaymentReservationFailedEvent
      │
      ▼
ReleaseInventoryCommand
      │
      ▼
InventoryReleasedEvent
      │
      ▼
OrderCancelled (Domain Event)
      │
      ▼
OrderCancelledEvent

Получив событие PaymentReservationFailedEvent, Saga переходит по ветви компенсации и отправляет команду ReleaseInventoryCommand.
После успешного освобождения товара Stock Service публикует событие InventoryReleasedEvent. 
Получив его, Order Service переводит заказ в состояние Cancelled, генерирует доменное событие OrderCancelled, которое затем преобразуется в интеграционное событие OrderCancelledEvent.

Типовые сценарии
В рамках процесса оформления заказа возможны следующие бизнес-сценарии:
1. Успешное выполнение — товар успешно зарезервирован, денежные средства успешно зарезервированы, заказ переводится в состояние Completed.
2. Ошибка резервирования товара — публикуется событие InventoryReservationFailedEvent, Saga переводит заказ в состояние Cancelled, компенсация не требуется, поскольку ни одна из последующих операций ещё не была выполнена.
3. Ошибка резервирования оплаты — после получения события PaymentReservationFailedEvent Saga отправляет команду ReleaseInventoryCommand, после завершения компенсации заказ переводится в состояние Cancelled.

Примечание.
Помимо бизнес-ошибок, Saga обрабатывает инфраструктурные ситуации: временную недоступность сервисов, таймауты, повторную доставку сообщений, повторное выполнение команд и другие сбои распределённой системы. Эти механизмы относятся к обеспечению надёжности обмена сообщениями и не изменяют бизнес-логику жизненного цикла заказа.

## 7.7 Обмен сообщениями
Взаимодействие между микросервисами осуществляется посредством обмена сообщениями через брокер сообщений.

Используются два основных типа сообщений:
- Commands (команды);
- Integration Events (интеграционные события).

Commands (Команды)
Команда представляет собой запрос на выполнение определённого действия.
Команда направляется конкретному сервису, который является единственным обработчиком данного сообщения (модель взаимодействия 1-к-1).
После обработки команды сервис не возвращает прямой ответ отправителю. Вместо этого он публикует одно из интеграционных событий, сообщающее о результате выполнения операции.

Примеры команд:
- ReserveInventoryCommand
- ReleaseInventoryCommand
- ReservePaymentCommand

Integration Events (Интеграционные события)
Интеграционное событие уведомляет другие микросервисы о произошедшем факте.
События публикуются по модели 1-ко-многим. Отправитель не знает и не должен знать, какие сервисы подпишутся на опубликованное событие.
Интеграционные события не предполагают получения ответа.

Примеры интеграционных событий:
- InventoryReservedEvent — товар успешно зарезервирован;
- InventoryReservationFailedEvent — резервирование товара завершилось ошибкой;
- InventoryReleasedEvent — ранее зарезервированный товар освобождён;
- PaymentReservedEvent — денежные средства успешно зарезервированы;
- PaymentReservationFailedEvent — резервирование денежных средств завершилось ошибкой;
- OrderCompletedEvent — заказ успешно завершён;
- OrderCancelledEvent — заказ отменён.

Одно интеграционное событие может одновременно обрабатываться несколькими микросервисами независимо друг от друга.

## 7.8 Domain Events и Integration Events
В системе используются два уровня событий: Domain Events и Integration Events.
Такое разделение позволяет отделить внутреннюю модель предметной области от взаимодействия между микросервисами.

Domain Events
Domain Event описывает изменение состояния предметной области внутри одного микросервиса.
Доменные события используются исключительно внутри сервиса и не публикуются во внешнюю систему.

Примеры доменных событий в Order Service:
OrderCreated
OrderCompleted
OrderCancelled

Доменные события позволяют реализовать внутреннюю бизнес-логику, не связывая её с механизмами обмена сообщениями между сервисами.

Integration Events
Integration Event предназначен для обмена информацией между микросервисами.
После обработки доменного события сервис при необходимости формирует и публикует соответствующее интеграционное событие.
Все интеграционные события являются частью проекта PizzaSaga.Contracts, который содержит общий контракт взаимодействия между сервисами.

Примеры:
InventoryReservedEvent
InventoryReservationFailedEvent
InventoryReleasedEvent
PaymentReservedEvent
PaymentReservationFailedEvent
OrderCompletedEvent
OrderCancelledEvent

Например, после перевода заказа в состояние Completed Order Service генерирует доменное событие OrderCompleted. Затем оно преобразуется в интеграционное событие OrderCompletedEvent, которое публикуется в брокер сообщений и становится доступным другим микросервисам.
Аналогично, после перевода заказа в состояние Cancelled создаётся доменное событие OrderCancelled, которое преобразуется в OrderCancelledEvent.
Такое разделение позволяет не раскрывать внутреннюю модель предметной области другим сервисам, а также упрощает развитие системы и снижает связанность между её компонентами.

## 7.9 Надёжная доставка сообщений (Outbox Pattern)
Для обеспечения согласованности между базой данных и брокером сообщений используется Transactional Outbox Pattern.
Во время обработки команды выполняется одна локальная транзакция.

В рамках этой транзакции:
сохраняются изменения бизнес-сущностей;
сохраняется запись в таблице Outbox.

После успешной фиксации транзакции специальный процесс доставки сообщений считывает записи из таблицы Outbox и публикует их в RabbitMQ.

Business Operation
        │
        ▼
Database Transaction
        │
        ├─────────────► Update Aggregate
        │
        └─────────────► Insert Outbox Message
                          │
                          ▼
                 Background Publisher
                          │
                          ▼
                     RabbitMQ

Такой подход предотвращает потерю сообщения между сохранением данных и их публикацией в брокер сообщений.

## 7.10 Идемпотентность обработчиков
RabbitMQ и MassTransit поддерживают повторную доставку сообщений.
Поэтому каждый обработчик должен быть идемпотентным.
Повторная обработка одного и того же сообщения не должна изменять результат выполнения операции.

Для обеспечения идемпотентности могут использоваться:
уникальный MessageId;
таблица обработанных сообщений;
проверка текущего состояния агрегата.

Выбор конкретного механизма зависит от сценария обработки.

## 7.11 CorrelationId и Saga
Каждый экземпляр Saga идентифицируется с помощью CorrelationId.
В Saga CorrelationId используется для поиска экземпляра State Machine, которому предназначено поступившее сообщение.

Этот идентификатор:
передаётся между сервисами;
сохраняется в состоянии Saga;
используется при публикации сообщений;
включается в журналы и трассировки.

Благодаря этому можно проследить выполнение всего бизнес-процесса от HTTP-запроса до завершения Saga.

## 7.12 Обработка ошибок
При возникновении ошибки применяется следующий порядок действий:
локальная транзакция откатывается;
сообщение не публикуется (если транзакция не завершилась);
при необходимости MassTransit выполняет повторную доставку сообщения;
если попытки исчерпаны, сообщение помещается в очередь ошибок;
После исчерпания попыток обработки Saga либо инициирует компенсирующую операцию, либо переводит бизнес-процесс в состояние Cancelled в соответствии с бизнес-логикой.

Такой механизм обеспечивает устойчивость системы к временным сбоям.

## 7.13 Принципы обмена сообщениями
При разработке сервисов соблюдаются следующие правила:
• сервисы не имеют прямых ссылок друг на друга;
• взаимодействие осуществляется только через PizzaSaga.Contracts;
• сообщения являются неизменяемыми;
• команды описывают намерение выполнить действие;
• Integration Events описывают уже произошедший факт;
• каждый сервис отвечает только за собственные данные;
• обработчики сообщений должны быть идемпотентными;
• Saga координирует выполнение бизнес-процесса, но не реализует бизнес-логику других сервисов.

## 7.14 Итог
Для координации распределённого бизнес-процесса в PizzaSaga используется оркестрируемая Saga на базе MassTransit State Machine.
Состояние выполнения бизнес-процесса хранится в экземпляре Saga, что позволяет безопасно продолжать выполнение после получения очередного сообщения.
Обмен сообщениями осуществляется через RabbitMQ с использованием команд и интеграционных событий.
Надёжная доставка сообщений обеспечивается Transactional Outbox Pattern, а корректность повторной обработки достигается за счёт идемпотентности обработчиков.
Такое сочетание паттернов позволяет реализовать согласованный бизнес-процесс без использования распределённых транзакций, сохраняя слабую связанность сервисов и высокую отказоустойчивость системы.

#8. Observability (наблюдаемость)
## 8.1 Назначение
Observability обеспечивает возможность отслеживать работу распределённой системы, анализировать причины ошибок и восстанавливать полный путь выполнения бизнес-операции через несколько микросервисов.

В проекте PizzaSaga наблюдаемость является одной из ключевых инфраструктурных возможностей, поскольку обработка заказа включает взаимодействие нескольких сервисов, обмен сообщениями через RabbitMQ и выполнение распределённого бизнес-процесса под управлением Saga.

Все сервисы используют единый подход к логированию, сбору метрик и распределённому трейсингу.

## 8.2 Реализация в PizzaSaga
Общая конфигурация наблюдаемости централизована в проекте PizzaSaga.ServiceDefaults, который подключается каждым сервисом.

ServiceDefaults автоматически настраивает:
OpenTelemetry;
структурированное логирование (Serilog);
сбор метрик;
распределённый трейсинг;
Health Checks;
передачу CorrelationId между сервисами.

Благодаря централизованной конфигурации все сервисы используют единые правила логирования, мониторинга и трассировки без дублирования инфраструктурного кода.

## 8.3 Используемые инструменты
В проекте используются следующие компоненты наблюдаемости.

Компонент				Назначение
Serilog					Структурированное логирование
OpenTelemetry			Сбор трейсов и метрик
.NET Aspire Dashboard	Просмотр логов, метрик и трейсов во время разработки
Prometheus				Сбор и хранение метрик
Grafana					Визуализация метрик
Jaeger					Просмотр распределённых трейсов

Во время локальной разработки основным инструментом диагностики является .NET Aspire Dashboard. Остальные компоненты демонстрируют типичный production-подход к наблюдаемости распределённых систем.

## 8.4 Логирование
Все сервисы используют структурированное логирование на базе Serilog.

В каждый лог автоматически включаются:
имя сервиса;
уровень сообщения;
CorrelationId;
TraceId.

Это позволяет объединять записи журналов различных сервисов в рамках одного бизнес-процесса и существенно упрощает диагностику ошибок.

## 8.5 Метрики
Метрики автоматически собираются OpenTelemetry и экспортируются в Prometheus.

В Grafana отображаются основные показатели работы системы:
количество HTTP-запросов;
время обработки запросов;
количество ошибок;
использование ресурсов приложения;
состояние сервисов.

Эти данные позволяют оценивать производительность системы и быстро обнаруживать возникающие проблемы.

## 8.6 Распределённый трейсинг
OpenTelemetry автоматически формирует распределённый трейс, охватывающий все этапы обработки запроса.

В трейс включаются:
HTTP-запросы;
обращения к PostgreSQL;
обмен сообщениями через RabbitMQ;
выполнение операций Saga.

Это позволяет проследить выполнение одной пользовательской операции от обращения к API Gateway до завершения распределённого бизнес-процесса.

## 8.7 Наблюдаемость Saga
Order Saga является центральным элементом распределённого бизнес-процесса, поэтому её выполнение полностью наблюдаемо.

Во время выполнения автоматически фиксируются:

запуск экземпляра Saga;
переходы между состояниями;
отправка команд;
получение интеграционных событий;
выполнение компенсирующих операций;
успешное или неуспешное завершение процесса.

Благодаря этому можно восстановить последовательность выполнения заказа и определить этап, на котором возникла ошибка.

## 8.8 CorrelationId и TraceId
Каждая пользовательская операция получает два идентификатора:

CorrelationId — связывает между собой все сообщения и журналы одного бизнес-процесса;
TraceId — используется OpenTelemetry для построения распределённого трейса.

Оба идентификатора автоматически передаются между сервисами через HTTP и RabbitMQ, благодаря чему весь путь обработки запроса можно проследить как в логах, так и в системе распределённого трейсинга.

## 8.9 Итог
Наблюдаемость в PizzaSaga реализована как единая инфраструктурная возможность, используемая всеми микросервисами.

Сочетание OpenTelemetry, Serilog, Prometheus, Grafana, Jaeger и .NET Aspire Dashboard позволяет:
анализировать работу сервисов в реальном времени;
диагностировать ошибки распределённых бизнес-процессов;
отслеживать выполнение Saga;
восстанавливать полный путь обработки пользовательского запроса.
