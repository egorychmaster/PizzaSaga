🚀 Sprint 0: Инфраструктурный MVP, API Gateway и Observability SDK
Цель: Развернуть базовый каркас распределенной системы с настроенным API Gateway, централизованной аутентификацией и сквозным OpenTelemetry-трейсингом для контроля CorrelationId с первого дня.

📋 Задачи:
1. Решение и оркестрация: Создать файл нового формата решений PizzaSaga.slnx и базовую структуру каталогов согласно architecture.md (src/Gateway, src/Services, src/Shared).

2. .NET Aspire: Настроить PizzaSaga.AppHost и PizzaSaga.ServiceDefaults. Подключить контейнеры PostgreSQL и RabbitMQ. Включить OpenTelemetry (Tracing, Metrics, Logs) в ServiceDefaults.

3. API Gateway (YARP): Создать проект PizzaSaga.Gateway. Настроить проксирование маршрутов /api/v1/orders/ на Order Service и /api/v1/auth/ на Auth Service. Реализовать middleware, которое обеспечивает CorrelationId propagation (извлечение/генерация и проброс в заголовках downstream-запросов). TraceId создаётся OpenTelemetry автоматически на базе Activity.
Создание кастомного Middleware/делегата для YARP, который проверяет наличие заголовка CorrelationId во входящем запросе. Если его нет — генерирует (Guid), если есть — сохраняет в AsyncLocal (или через Activity.Current.SetTag()) и принудительно пробрасывает во все downstream-запросы.
YARP транслирует токен, а сервисы валидируют его через единый метод расширения из ServiceDefaults (подход Zero Trust).

4. Auth Service: Создать каркас сервиса. Реализовать эндпоинт POST /api/v1/auth/login (выдача JWT access_token и refresh_token согласно спецификации api.md). Проверку учетных данных выполнять на основе предопределенного пользователя, логин и пароль зашиты в конфигурации appsettings.json. В файле конфигурации appsettings.json сервиса Auth Service структуру тестового пользователя например представить так:
{
  "TestUser": {
    "Username": "test_user",
    "PasswordHash": "$2a$11$...", 
    "UserId": "2fbb95d2-0672-4b2f-b3fd-447d2a02d5f8"
  }
}
Зафиксировав конкретный UserId (GUID) для этого тестового пользователя прямо в конфигурации. Это позволит в будущем (в E2E-тестах или при ручном тестировании через Swagger) точно знать, какой именно идентификатор пользователя (sub claim) зашит в твоем JWT-токене, и легко проверять, что заказы в БД Order Service создаются именно на этот CustomerId.

5. Order Service (Начальный срез): Реализовать подход Zero Trust для аутентификации. Настроить в проекте PizzaSaga.Services.Order.Api независимую валидацию JWT-токенов с помощью стандартного пакета Microsoft.AspNetCore.Authentication.JwtBearer (используя общие ключи/настройки, вынесенные в ServiceDefaults). API Gateway будет просто проксировать оригинальный Bearer-токен дальше в downstream. Добавить базовый эндпоинт POST /api/v1/orders, принимающий уже аутентифицированный запрос (с валидным токеном от Auth) и возвращающий 201 Accepted со структурой ответа согласно api.md.

6. Инфраструктурная обвязка: Настроить Health Checks (/health/live, /health/ready). Инфраструктурная обвязка, сквозное логирование и обработка ошибок:
- Serilog Integration: Интегрировать Serilog во все микросервисы и API Gateway. Настроить структурированное логирование (Structured Logging) в консоль в формате JSON. Добавить в конфигурацию логгера обязательное обогащение контекста свойствами TraceId, SpanId и CorrelationId (через Serilog.Enrichers.Span / Serilog.Formatting.Compact).
- Глобальный ExceptionHandler: Реализовать централизованный обработчик исключений, используя современный интерфейс IExceptionHandler (введенный в .NET 8/9/10), во всех создаваемых сервисах.
- Конфигурация ProblemDetails: Настроить глобальный ProblemDetails (builder.Services.AddProblemDetails()) согласно спецификации RFC 9457 и требованиям api.md. Любое необработанное исключение (500 Internal Server Error) на уровне инфраструктуры должно перехватываться, логироваться через Serilog с сохранением TraceId, а клиенту должен возвращаться чистый JSON-ответ, скрывающий внутренние детали реализации бэкенда, но содержащий публичный traceId для разбора инцидентов.

7. Автоматические миграции и Seed данных (Order Service): Реализовать стратегию автоматического наката миграций для базы данных Order Service через расширение .NET Aspire (DbContext.ApplyMigrations()) или с помощью IHostedService (например, DatabaseMigrationHostedService), гарантирующего вызов Database.Migrate() строго до старта веб-хоста.

8. Стандартизация OpenAPI и настройка маршрутизации схем в YARP
Описание: Настроить генерацию OpenAPI-схем в PizzaSaga.ServiceDefaults, чтобы каждый новый микросервис автоматически получал настроенный Swagger.
Технические требования:
- В проекте PizzaSaga.ServiceDefaults создать метод расширения AddProjectOpenApi() (используя современный пакет Microsoft.AspNetCore.OpenApi для .NET 8/9/10 или Swashbuckle.AspNetCore). Настроить генерацию JWT-авторизации (Security Definitions), чтобы в Swagger UI можно было вставлять Bearer-токен.
- Подключить этот метод во все создаваемые сервисы (Auth, Order).
- Маршрутизация в YARP Gateway: В файле конфигурации PizzaSaga.Gateway настроить проксирование путей Swagger. Внешний запрос /swagger/auth/v1/swagger.json проксируется на внутренний /swagger/v1/swagger.json Auth-сервиса, а /swagger/order/v1/swagger.json — на Order-сервис.
- Настроить локальный Swagger UI в самом API Gateway, который просто ссылается на статические URL-адреса схем сервисов, проксируемых через него (но не агрегирует их физически в одну схему).

✅ Критерии приемки:
- Запуск .NET Aspire поднимает систему.
- Переход по адресу Gateway http://localhost:gateway-port/swagger открывает страницу, где в выпадающем списке (Select a definition) можно выбрать контракт любого сервиса: "Auth Service API", "Order Service API".
- Запросы к эндпоинтам успешно выполняются прямо из этого интерфейса через прокси Gateway.
- Запрос POST /api/v1/auth/login через Gateway возвращает валидный JWT.
- Запрос POST /api/v1/orders с валидным JWT возвращает 202 Accepted. В .NET Aspire Dashboard виден сквозной трейс: Gateway -> Order.Api.

⚠️ Технические риски: Настройка конфигурации маршрутизации YARP и генерации заголовков.


🧱 Sprint 1: Domain Model, CQRS и Vertical Slice Architecture в Order Service
Цель: Реализовать полноценную доменную логику Order Service по принципам Tactical DDD, изолировать домен от инфраструктуры с помощью CQRS (Mediator) и настроить строгое соответствие спецификации ошибок.

📋 Задачи:
1. Domain Layer (Order.Domain): Создать изолированный слой. 
Вместо примитивных типов определить бизнес-концепции как Value Objects с закрытыми конструкторами и фабричными методами (парсерами):
- Money (содержит Amount и Currency, парсит сырой decimal и string, гарантируя неотрицательность и соответствие ISO кодам).
- CustomerIdentity (инкапсулирует Guid пользователя, делая невозможным создание пустого или дефолтного ID).
- PizzaQuantity (целое число от 1 до 10, парсит int, исключая нулевой или отрицательный заказ).
1.1 Реализовать доменное событие OrderCreatedDomainEvent.

2. Application Layer (Order.Application): Организовать структуру по Vertical Slices (папка Features/Orders/CreateOrder). Реализовать CreateOrderCommand, CreateOrderCommandHandler и входные DTO.
- Валидация синтаксиса на входе: Настроить FluentValidation строго для базовой проверки структуры запроса (синтаксический анализ: строка не null, массив позиций заказа не пустой, GUID пришел в правильном строковом формате).
- Парсинг на границе домена: Внутри CreateOrderCommandHandler (или специализированного фабричного метода команды) сырые типы из DTO передаются в фабрики Value Objects домена (Money.Create(), PizzaQuantity.Create()). Обработчик не проверяет флаг IsValid, он сразу пытается распарсить (создать) объекты. Если фабричный метод возвращает ошибку/исключение, пайплайн прерывается. CreateOrderCommand должна оперировать уже валидными типами домена, а не сырыми decimal или string.
2.1. Безопасное извлечение идентичности (Secure Identity Propagation):
- Полностью удалить поле CustomerId из публичного JSON-тела запроса CreateOrderRequest (клиент не должен иметь возможности подделать ID пользователя).
- Настроить аутентификацию в Order.Api (валидация JWT-токенов, выпущенных Auth-сервисом).
- В минимальном API / контроллере эндпоинта POST /api/v1/orders извлекать UserId из токена авторизованного пользователя через HttpContext.User.FindFirst(ClaimTypes.NameIdentifier) (или использовать аналогичный claim sub).
- Сконструировать CreateOrderCommand, обогатив её извлеченным UserId (передав его в конструктор команды как CustomerId), и только после этого отправлять команду в Mediator.
2.2 Использовать FluentValidation строго для синтаксического контроля DTO на уровне API, а бизнес-инварианты проверять исключительно при конструировании Value Objects (концепция Parse, Don't Validate).

3. Pipeline Behaviors: Интегрировать компиляционный Mediator. Реализовать следующие Behaviors:
- ValidationBehavior: Выполняет автоматический запуск FluentValidation. Его зона ответственности — исключительно синтаксический контроль входного DTO. Если в запросе не хватает обязательного поля или нарушена структура JSON — пайплайн падает с ValidationException до открытия транзакции. Он не проверяет бизнес-ограничения (например, что цена больше нуля или пицц не больше 10).
- TransactionBehavior: Автоматическое управление транзакциями PostgreSQL для команд (ICommand), использующее IExecutionStrategy для transient-ошибок.
	
Маркерные интерфейсы (CQRS-разделение): Разделить запросы на уровне типов: ввести интерфейсы-маркеры ICommand<TResponse> (для изменения состояния) и IQuery<TResponse> (для чтения).
Реализовать три ключевых Behavior:
LoggingBehavior — структурированное логирование входящих запросов и времени их выполнения.
ValidationBehavior — автоматический запуск FluentValidation перед выполнением команд. Применяется только к ICommand.
TransactionBehavior — автоматическое открытие транзакции БД PostgreSQL. Применяется строго к командам (ICommand), реализующим транзакционную логику.
Технические требования к TransactionBehavior:
Использовать стратегию повторных попыток EF Core (IExecutionStrategy) для обработки transient-ошибок (кратковременных сбоев сети СУБД) при старте транзакции.
Обеспечить автоматический вызов SaveChangesAsync() и коммит транзакции только при успешном завершении обработчика (Handler). При возникновении исключения — автоматический Rollback.
Исключить применение TransactionBehavior для Query-запросов (чтения), чтобы избежать лишних блокировок в БД и повысить пропускную способность.

4. Infrastructure Layer (Order.Infrastructure): Настроить EF Core с PostgreSQL. Настроить маппинг агрегата Order без утечки инфраструктурных зависимостей в домен. Настроить Optimistic Concurrency с использованием явного свойства Version (через .IsConcurrencyToken() в Fluent API EF Core), инкрементируемого при каждом изменении агрегата, как это определено в архитектуре.

5. Централизованная обработка ошибок: Реализовать IExceptionHandler, преобразующий исключения в формат RFC 9457 (ProblemDetails):
- ValidationException (от ValidationBehavior) -> преобразуется в 400 Bad Request с типом ошибки "Нарушение структуры запроса" (синтаксис).
- DomainParsingException или DomainException (возникающий, когда доменный фабричный метод не смог распарсить сырые данные, например NegativeMoneyException) -> преобразуется в 400 Bad Request с типом ошибки "Нарушение бизнес-правил" (семантика).
- DbUpdateConcurrencyException -> преобразуется в 409 Conflict.

6. HTTP Idempotency (PostgreSQL-based): Реализовать IdempotencyFilter (или Middleware) для эндпоинта создания заказа.
- Создать таблицу IdempotencyKeys в БД Order Service (колонки: IdempotencyKey (primary key), ResponseStatusCode, ResponseBody, CreatedAt).
- При получении запроса с заголовком Idempotency-Key выполнять атомарную проверку и вставку ключа в БД в рамках единой транзакции с созданием заказа.
- Если ключ уже существует — прерывать пайплайн и возвращать сохраненный в таблице ответ (201 Created / ResponseBody) без повторного запуска доменной логики. Если заказ в процессе обработки — возвращать соответствующий промежуточный статус согласно api.md.
- При повторном использовании существующего Idempotency-Key необходимо сравнить hash тела запроса. Если тело отличается — вернуть 409 Conflict.
- Добавить в задачу обязательную обработку DbUpdateException (дубликат ключа) внутри IdempotencyFilter. Если поймали ошибку уникального индекса для IdempotencyKey — нужно:
- Отменить текущую транзакцию.
- Сделать повторный запрос в таблицу идемпотентности, чтобы забрать тот ответ, который сгенерировала параллельная (успешная) транзакция, либо вернуть статус 409 Conflict / 423 Locked (если первый запрос еще обрабатывается).

✅ Критерии приемки:
- Эндпоинт POST /api/v1/orders вызывает IMediator.Send(). Напрямую DbContext в контроллерах/минималках не используется.
- При отправке некорректных данных API возвращает 400 Bad Request в формате ProblemDetails со списком нарушенных бизнес-правил.
- Повторный запрос с тем же Idempotency-Key не создает новую запись в БД, а возвращает 201 Created из кэша/таблицы идемпотентности.


🔗 Sprint 2: Saga Orchestration & Messaging Skeleton (Скелет распределенной системы)
Цель: Развернуть инфраструктуру обмена сообщениями, запустить Transactional Outbox паттерн и реализовать стейт-машину Саги, взаимодействующую со «скелетами» зависимых сервисов.

- Перенести в него все общие контракты интеграционных сообщений (команд и событий) MassTransit.

Но при получении сообщения потребителем (Consumer) в Stock Service или Payment Service, сырые данные сразу же парсятся во внутренние Value Objects этих доменов (например, Quantity преобразуется в InventoryStockQuantity в Stock-сервисе). Консьюмеры не работают с сырыми int или string в своей бизнес-логике (концепция Parse, Don't Validate).

- Внутреннее состояние Cancelled маппится в публичный статус Cancelled.
- Настроить персистентность инстансов Саги в базе данных Order через EF Core (Saga Repository).
- OrderStateMachine при переходе в терминальные состояния (Completed, Cancelled) публикует локальную команду MassTransit (или доменное событие), которую слушает консьюмер в Order Service и обновляет поле Status у сущности Order. Либо Сага напрямую использует DbContext для обновления заказа (что допустимо, если они делят одну БД, но первый вариант чище).

5. Создание Заглушек Потребителей (Consumers):

6. Transactional Outbox & Inbox для всех сервисов:

- Реализовать диспетчеризацию Domain Events в Integration Events перед публикацией через Outbox.
6.1 Реализовать сохранение доменных событий в EF Core DbContext перегрузкой метода SaveChangesAsync. События пишутся в таблицу OutboxMessages в той же транзакции. MassTransit Outbox автоматически подхватит их и опубликует в RabbitMQ как интеграционные сообщения после успешного коммита.

✅ Критерии приемки:
- При вызове POST /api/v1/orders заказ сохраняется в БД Order, а в таблицу OutboxMessages пишется событие. Фоновый воркер MassTransit отправляет его в RabbitMQ.

🛡️ Sprint 3: Реальная бизнес-логика Саги, Конкурентность и Компенсации (Резильентность)
Цель: Заменить заглушки в Stock и Payment сервисах на реальную бизнес-логику работы с данными, реализовать обработку отказов и логику компенсирующих транзакций Саги.

📋 Задачи:
1. Бизнес-идемпотентность на уровне Домена (Idempotent Consumers / Business Level):
- Внедрить проверку состояния доменного агрегата/сущности перед выполнением мутирующих операций в Stock и Payment сервисах (state-based idempotency).
- Для Stock Service: перед резервированием товара проверять, не был ли этот конкретный OrderId уже успешно обработан в данной системе (например, через связь один-к-одному «Заказ-Резерв» или через сохранение бизнес-билетов операций). Повторная обработка команды на резерв для уже зарезервированного заказа должна быть идемпотентным no-op (просто возвращать успешный ответ, не списывая остатки повторно).
- Для Payment Service: перед транзакцией проверять статус оплаты этого OrderId в БД. Если оплата уже прошла успешно (или завершилась отказом) — повторная команда на авторизацию платежа не должна повторно дергать логику эквайринга, а должна сразу возвращать зафиксированный ранее доменный результат.

2. Бизнес-логика Stock Service: Реализовать сущность Inventory. При получении команды ReserveInventoryCommand консьюмер уменьшает количество товара в БД Stock. Использовать Optimistic Concurrency на уровне сущности товара для защиты от состояния гонки (Race Condition) при одновременных заказах. Если товара недостаточно — публиковать InventoryReservationFailed.
При возникновении DbUpdateConcurrencyException в консьюмере, транзакция падает, и сообщение должно уходить на Retry (настраивается через MassTransit UseMessageRetry). Это инфраструктурная transient-ошибка, а не бизнес-отказ. Бизнес-отказ (InventoryReservationFailedEvent) публикуется только в том случае, если физически на складе StockLevel меньше запрошенного.
Stock Service использует "Parse, Don't Validate" для управления остатками: сущность Inventory (агрегат) инкапсулирует бизнес-метод ReduceStock(Quantity quantity). Если запрашиваемое количество превышает остаток, агрегат генерирует ошибку бизнес-логики (DomainException), что приводит к публикации InventoryReservationFailedEvent.


3. Бизнес-логика Payment Service: Реализовать симуляцию списания средств. Добавить в конфигурацию сервиса управляемый флаг SimulateFailure. Если флаг активен — консьюмер публикует PaymentAuthorizationFailed, имитируя ошибку банка.

4. Компенсационные сценарии Саги: Расширить OrderStateMachine.
- Если Stock прислал InventoryReservationFailedEvent -> перевести заказ в статус Cancelled.
- Если Payment прислал PaymentAuthorizationFailedEvent -> Сага должна отправить в Stock компенсирующую команду ReleaseInventoryCommand (расформировать резерв), и только после этого перевести заказ в статус Cancelled.
 
5. Реализовать ReleaseInventoryConsumer:
    • обработать команду ReleaseInventoryCommand;
    • освободить ранее зарезервированный товар;
    • опубликовать InventoryReleasedEvent;
    • обеспечить идемпотентность обработки.

6. Политики устойчивости (Resilience) и DLQ: Настроить MassTransit Retries (политика повторных попыток при дедлоках БД или временных сбоях сети). Настроить Dead Letter Queue (DLQ) для сообщений, которые не удалось обработать после всех повторов.

✅ Критерии приемки:
- При успешном прохождении (флаг сбоя отключен) остатки в БД Stock уменьшаются, Сага переходит в Completed.
- При симуляции ошибки оплаты Сага автоматически посылает команду отмены резерва в Stock. В БД Stock остатки возвращаются в исходное состояние, статус заказа в БД Order становится Cancelled.
- При возникновении исключений в консьюмере MassTransit делает 3 попытки, после чего перемещает сообщение в очередь *_error (DLQ).
